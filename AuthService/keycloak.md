# Keycloak Configuration Guide (Microservices Ecosystem)

This document outlines the step-by-step setup required inside the Keycloak Admin Console to support our API Gateway, custom Authentication Service, and individual Resource Server microservices.

---

## 1. Realm Configuration

To isolate our application ecosystem from the default master realm, we must first provision a dedicated realm.

1. Open the Keycloak Admin Console (`http://localhost:8080`).
2. Hover over the top-left dropdown menu (which defaults to `master`) and click **Create Realm**.
3. Set the **Realm name** to exactly: `oauth-test`
4. Click **Create**.

---

## 2. Master Client Configuration (`oauth-test-service-client`)

This is the primary administrative client configuration used by the API Gateway and the custom `auth-service` to execute server-to-server calls, handle user logins, and dynamically provision client tenants.

### Step A: Client Creation
1. Navigate to **Clients** in the left sidebar menu -> Click **Create client**.
2. **Client type:** `OpenID Connect`
3. **Client ID:** `oauth-test-service-client`
4. **Name:** `Master Service Client`
5. Click **Next**.

### Step B: Capability Config
Toggle the settings precisely as follows to enable all required OAuth2 grant types:
* **Client Authentication:** `ON` *(Changes the client type to Confidential, generating a secure secret)*
* **Authorization:** `OFF`
* **Authentication Flow:**
    * [x] Standard flow *(Enables Authorization Code Flow for UI redirect logins)*
    * [x] Direct access grants *(Enables Password Grant Flow for test user accounts)*
    * [x] Service accounts roles *(Enables Client Credentials Flow for server-to-server admin tasks)*
6. Click **Save**.

### Step C: Access Settings & Redirects
Scroll down to the **Access settings** section and configure the white-listed redirection constraints:
* **Valid redirect URIs:** `http://localhost:8084/login/oauth2/code/keycloak` *(Matches Spring Security's default registration naming conventions)*
* **Web origins:** `http://localhost:8084` *(Allows incoming requests from the Gateway)*
* Click **Save**.

### Step D: Admin Rights Assignment (`403 Forbidden` Fix)
By default, clients cannot manage other clients. To allow this master client to programmatically generate *new* client configurations via Keycloak's Admin REST API:
1. Switch to the **Service accounts roles** tab at the top of the client page.
2. Click **Assign role**.
3. Change the top dropdown filter from *"Filter by realms"* to **"Filter by clients"**.
4. Type **`realm-management`** in the search box and select the resource instance.
5. Check the boxes next to **`manage-clients`** and **`view-clients`**.
6. Click **Assign**.

### Step E: Retrieve Client Secret
1. Switch to the **Credentials** tab at the top of the client page.
2. Copy the string value next to **Client Secret**.
3. Paste this value into your `api-gateway` and `auth-service` `application.properties` configurations under:
   `spring.security.oauth2.client.registration.keycloak.client-secret`

---

## 3. Global Roles Definition

We need to define global roles within the realm so that they can be packaged into JWT claims for microservice verification.

1. Navigate to **Realm roles** in the left navigation sidebar.
2. Click **Create role**.
3. **Role name:** `admin`
4. Click **Save**.
5. Click **Create role** again.
6. **Role name:** `user`
7. Click **Save**.

---

## 4. Dynamic Token Adjustments (Client Scopes Mapping)

When the `auth-service` programmatically provisions a client (B2B tenant), Keycloak must be instructed to map roles and scopes directly into that client's issued access token payload.

### Step A: Assign Roles to Client Service Account
1. Go to **Clients** -> Select your programmatically created client ID (e.g., `my-microservice-tenant-a4f21`).
2. Switch to the **Service accounts roles** tab.
3. Click **Assign role** and select the appropriate system permission (e.g., `admin`).

### Step B: Attach Client Scopes to Token
1. Inside that same client configuration page, navigate to the **Client Scopes** tab.
2. Click **Add client scope** -> **From existing client scopes**.
3. Select `profile`, `email`, and `roles`.
4. Click **Add** and choose **Default**.

*Note: This guarantees that when the tenant invokes the client credentials grant, the final JWT payload will include the `realm_access.roles` block required by your downstream microservices.*

## 6. Common Infrastructure Adjustments

### 1. `ECONNREFUSED ::1:8080` (IPv6 Resolution Bug)
If your Spring applications fail to connect to Keycloak during startup, your OS is trying to map `localhost` to the IPv6 loopback address (`::1`), while Keycloak is listening exclusively on IPv4 (`127.0.0.1`).
* **The Fix:** Explicitly change `localhost` to `127.0.0.1` in your microservice application files:
  ```properties
  spring.security.oauth2.resourceserver.jwt.issuer-uri=[http://127.0.0.1:8080/realms/oauth-test](http://127.0.0.1:8080/realms/oauth-test)
